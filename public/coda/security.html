Security

agent がチームの一員のように実務を担いたい。その目的と安全性を両立する設計。

01

Objectives

  • 自律性—人間の介入なしに動ける
  • 実効性—本物のアクセス権で本物の仕事ができる
  • 共有性—チーム全員で同じ agent を使える
02

Concepts

これらの目的と安全性を両立させるのがセキュリティの役割。

  • シンプルな構造—3 Levels + 2 Runtimes + 4 Boundaries
  • 共有の範囲 = 権限の範囲—誰と共有するかがそのまま権限のスコープを決める
  • cap 内の自由—境界の内側では制限しない
  • メカニカルな境界—信頼やルールではなく技術的メカニズムで壁を作る
  • 変更への非同期の人間の介入—agent は自律的に動けるが、変更の反映は PR 経由で人間がレビューする
03

Overview

Levels が情報の機密度を分け、Runtimes が実行経路ごとの到達範囲を決め、Boundaries がその境界を技術的に強制する。

external member admin
Network+Change
Process+Change+Auth
bot
local

外側から external・member・admin。内側ほど機密性が高い。bot は member の中で動き、admin には届かない。local は admin まで届く。ゾーン間の破線が Boundary — 越境を防ぐ仕組み。

04

3 Levels

external アクセス不可
member Slack public チャンネルにいる全員が見ていい・操作していい範囲。内側は全て開放
admin keys 管理、workflow 設定、通信先ホワイトリスト、Slack App 管理

member / admin の実質的な壁は Google Account による KMS 復号権限。credential は SOPS + GCP KMS で暗号化されており、復号できる Account が境界を決める。

05

2 Runtimes

bot 共通のサービスアカウントで認証。member に cap。Slack と GitHub の両方がこの経路で動く。GitHub App トークン、チャンネルホワイトリスト、credential ホワイトリスト — 全て member 相当にスコープ
local 個人の Google Account で認証。KMS 復号権限があれば admin になりうる。実行者の権限がそのまま agent の権限になる。意図的な設計

唯一の例外は codatum-cloudflare-public への書き込み。公開静的コンテンツのため member 境界の外に出うるが、リポジトリを分離することで coda 本体への書き込み権限では変更できない設計にしている。

06

4 Boundaries

Process credential の実値は別プロセスのメモリにだけ存在する。agent は実値に触れない
Network 許可ホスト以外への通信を遮断。内部データの外部送信を防ぐ
Change main への直接 push は禁止。変更は PR 経由で人間がレビュー
Auth Google Account の認可が壁。credential の暗号化・復号を制御する

Local ではこのうち Process と Network が適用されない。credential に平文でアクセスでき、通信先の制限もない。
